Most backends assume a human in a dashboard. GrowFoundry assumes a coding agent at a terminal. The products (Database, Auth, Storage, and the rest) are the building blocks; the primitives on this page are how an agent operates them: as files it can edit, branches it can test on, and a backend it can diagnose and fix on its own.

</Note>

## Why it matters

A person and an agent want different things from a backend. A person wants a UI to click. An agent wants a stable text interface it can drive, read back, and reason about. That shows up everywhere in GrowFoundry.

Schema changes are [migrations](/core-concepts/database/migrations) in your repo, and project config is an [`growfoundry.toml`](/agent-native/config-as-code) file, so the agent edits text, commits it, and reviews it in a PR instead of clicking through forms. When it wants to try something risky, it spins up a [backend branch](/agent-native/branching), runs the change against a copy, and throws the branch away if it goes wrong. When something looks off, it fetches [diagnostics and advisor findings](/agent-native/diagnostics) directly with `npx @growfoundry/cli diagnose`, no dashboard in the loop, and applies the fix itself. That last part is also how the backend gets more secure: the agent reads a security finding like a permissive RLS policy and remediates it on its own, instead of waiting for a human to remember to check. And it reads current schemas, logs, and metadata straight from the backend with `npx @growfoundry/cli metadata`, so it works from real state rather than guessing.

## The loop

These chain together. A session usually goes:

1. Read the current state with `npx @growfoundry/cli metadata`.
2. Branch the backend, write a [migration](/core-concepts/database/migrations) and check what is pending with `npx @growfoundry/cli db migrations list`, or edit `growfoundry.toml` and preview the config diff with `npx @growfoundry/cli config plan`.
3. Apply it with `npx @growfoundry/cli db migrations up --all` or `config apply`, against the branch first, then the parent.
4. Run `npx @growfoundry/cli diagnose` to check advisor findings and error logs, and ask `diagnose --ai` to interpret them.
5. Apply the remediation, re-run diagnose, and merge the branch.
